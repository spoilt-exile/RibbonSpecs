Система обробки інформаційних повідомлень "Стрічка"

"Стрічка" - це новітня система обробки повідомлень для використання у інформаційних агентствах та інших виданнях. Основні функції системи полягають у структуризації повідомлень за технологічними напрямками/директоріями. Система дозволяє випускати, редагувати та видаляти повідомлення на інформаційних напрямках. Також повідомлення можливо помічати тегами. Доступ до напрямків контролюється гнучкою системою доступу. "Стрічка" цілком написана на мові програмування Java (з використанням OpenJDK 7). Код системи розповсюджується на умовах ліцензії GNU GPLv2.

Список програм з пакету системи "Стрічка":
  • RibbonServer - сервер системи [РЕЛІЗ];
  • RibbonProber - дебагер мережевого протоколу [СПЕЦИФІКАЦІЯ];
  • RibbonLastMessage - програма перегляду останнього повідомлення [СПЕЦИФІКАЦІЯ];
  • RibbonClient - програма-клієнт [СПЕЦИФІКАЦІЯ] [РЕЛІЗ];
  • RibbonWeb - веб-інтерфейс до системи [СПЕЦИФІКАЦІЯ];
  • JTVProg - подготовка та випуск телепрограми до системи [СПЕЦИФІКАЦІЯ];
Список бібліотек:
  • libRibbonData - бібліотека структур даних [СПЕЦИФІКАЦІЯ];
  • libRibbonApp - бібліотека ядра програм системи "Стрічка" [СПЕЦИФІКАЦІЯ];
  • libRibbonIO - бібліотека вводу/виводу повідомлень [СПЕЦИФІКАЦІЯ];
Список модулів імпорту:
  • ImportPlain - модуль імпорту текстових файлів [СПЕЦИФІКАЦІЯ];
  • ImportMail - модуль імпорту електронної пошти [СПЕЦИФІКАЦІЯ];
Список модулів експорту:
  • ExportPlain - модуль експорту текстових файлів [СПЕЦИФІКАЦІЯ];
  • ExportMail - модуль експорту електронної пошти [СПЕЦИФІКАЦІЯ];
Випущено версію a2.1. Розробка припинена.

Нумерація версій

Більшість компонентів системи використовують спільний ідентефікатор версії. Для альфа випусків: аХ; для нестабільних випусків: bX і для стабільних: RX (де X - це номер версії).

Мінорні випуски окремих компонентів отримують другу цифру: наприклад, R2.1. Усі програми у межах одного мажорного релізу мають гарантовану сумісніть між собою. Випуски для розробки чи тестування отримують ще ідентифікатор тестового зрізу: наприклад R2.2 slice-7.

Модулі імпорту і експорту не залежать від реалізації протоколу, тому вони використовують іншу нумерацію версії. Номер версії модуля залежить від версії його специфікації і статусу його готовності. Для помічання готовності модулю перед номером версії встановлюється спеціальний префікс. Всього їх чотири: vx (експериментальна реалізація специфікації), vt (тестова реалізація, відповідає специфікації), vs (стабільна реалізація) і vp (промислова реалізація яка пройшла тестування). Приклад ідентифікатора версії модуля: vx.1 (експериментальна реалізація першої версії специфікаці) чи vp.2 (промислова реалізація другої версії специфікації). Ідентифікатор мінорної версії модуля: vs.2.3 (стабільна реалізація другої версії специфікації з третьою поправкою).

Відомі обмеження поточної реалізації системи:

Далі надається список відомих обмеженнь і помилок у поточної реалізації системи. Ці помилки не будуть виправлені у поточному випуску а2 і відкладені на майбутнє.
  • Загальне: протокол адміністратора не описано у специфікації;
  • Загальне: вразливий протокол (користувач може скомпрометувати систему написавши команду END: + БУДЬ_ЯКА_КОМАНДА у повідомленні);
  • Загальне: поточна реалізація libRibbonIO не дозволяэ поновити експорт повідомлення при помилці;
  • Сервер: система не може змінювати свої налаштування без перезавантаження;
  • Клієнт: можливі помилки з синхронізацією відправлення повідомлень через мережу (не перевірялось);
  • Клієнт: немає навігації за тегами;
  • Веб-інтерфейс: немає окремого статусу повідомлення для поновлення спроби редагування;
  • Веб-інтерфейс: немає можливості переглядати напрямки;
  • Веб-інтерфейс: немає механізму сповіщень для користувача;
